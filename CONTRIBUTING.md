# コントリビューションガイド

empowered-api-specへのコントリビューションをご検討いただき、ありがとうございます！このドキュメントでは、プロジェクトへの貢献方法について説明します。

## 貢献できる内容

以下のような貢献を歓迎します：

1. **新しいプロンプトテンプレートの追加**
   - 新しいAPIタイプや仕様書形式のテンプレート
   - より特化した用途のプロンプト

2. **既存テンプレートの改善**
   - より詳細な説明の追加
   - 実用的な例の追加
   - わかりやすさの向上

3. **使用例の追加**
   - 実際のプロジェクトでの使用例
   - 特定の業界やドメインでの例

4. **ドキュメントの改善**
   - 誤字脱字の修正
   - より明確な説明
   - 英語版ドキュメントの追加

5. **バグ報告**
   - テンプレートの問題点
   - ドキュメントの不明瞭な点

## 貢献の手順

### 1. Issueの作成（推奨）

大きな変更を行う前に、Issueを作成して提案内容を共有することをお勧めします。
これにより、作業の重複を避け、方向性について事前に議論できます。

### 2. リポジトリのフォーク

```bash
# GitHubでリポジトリをフォーク後、ローカルにクローン
git clone https://github.com/YOUR_USERNAME/empowered-api-spec.git
cd empowered-api-spec
```

### 3. ブランチの作成

```bash
# 作業用のブランチを作成
git checkout -b feature/your-feature-name
```

ブランチ名の規則：
- `feature/` - 新機能追加
- `fix/` - バグ修正
- `docs/` - ドキュメント更新
- `refactor/` - リファクタリング

### 4. 変更の実施

#### 新しいテンプレートを追加する場合

1. `prompts/templates/` に新しい `.md` ファイルを作成
2. 以下の構造を含めてください：
   - 明確なタイトルと説明
   - `[...]` 形式のプレースホルダー
   - 各セクションの説明
   - 「生成してほしい内容」セクション

3. `prompts/README.md` に新しいテンプレートの説明を追加

4. `prompts/examples/` に使用例を追加（推奨）

#### 使用例を追加する場合

1. `prompts/examples/` に新しい `.md` ファイルを作成
2. テンプレートを基に、具体的な情報を記入した完全な例を作成
3. ファイル名は内容がわかりやすい名前にする（例：`product-catalog-api.md`）

#### ドキュメントを更新する場合

1. 該当するドキュメントファイルを編集
2. 日本語で記述（英語版は別途作成可）
3. 明確で簡潔な文章を心がける

### 5. 変更のコミット

```bash
# 変更をステージング
git add .

# コミット（わかりやすいメッセージで）
git commit -m "Add: 商品カタログAPI用のプロンプトテンプレート"
```

コミットメッセージの規則：
- `Add:` - 新規追加
- `Update:` - 更新
- `Fix:` - 修正
- `Docs:` - ドキュメント更新

### 6. プッシュとPull Request

```bash
# フォークしたリポジトリにプッシュ
git push origin feature/your-feature-name
```

GitHubでPull Requestを作成：
1. フォークしたリポジトリのページに移動
2. "Pull Request" ボタンをクリック
3. わかりやすいタイトルと説明を記入
4. 変更内容と目的を明確に説明

## Pull Requestのガイドライン

### タイトル

明確で簡潔なタイトルをつけてください。

例：
- ✅ `Add: GraphQL API用のプロンプトテンプレート`
- ✅ `Fix: OpenAPIテンプレートの認証セクション修正`
- ✅ `Docs: usage-guideに具体例を追加`
- ❌ `update`
- ❌ `修正`

### 説明

以下の情報を含めてください：
- **目的**: なぜこの変更が必要か
- **変更内容**: 何を変更したか
- **影響範囲**: どのファイルに影響するか
- **テスト方法**: どのように検証したか（該当する場合）

### チェックリスト

Pull Request作成時に以下を確認してください：
- [ ] 変更内容が明確で最小限である
- [ ] 新しいテンプレートには使用例がある
- [ ] ドキュメントが更新されている（該当する場合）
- [ ] コミットメッセージがわかりやすい
- [ ] 既存のファイル構造に従っている

## テンプレート作成のベストプラクティス

### 1. 明確な構造

テンプレートは以下の構造を持つことが推奨されます：

```markdown
# [テンプレート名]

[テンプレートの説明と用途]

## [セクション1名]
[説明とプレースホルダー]

## [セクション2名]
[説明とプレースホルダー]

## 生成してほしい内容
[AIエージェントに何を生成してほしいか明確に記載]
```

### 2. プレースホルダーの使用

ユーザーが置き換えるべき部分は `[...]` で囲みます：

```markdown
- **API名**: [APIの名前を記入]
- **バージョン**: [バージョン番号を記入]
```

### 3. 説明の追加

各セクションには、何を記入すべきか明確な説明を含めます：

```markdown
### エンドポイント1
- **メソッド**: [GET/POST/PUT/DELETE]
- **パス**: [エンドポイントのパス]
- **説明**: [エンドポイントの機能説明]
```

### 4. 実用的な例

可能な限り、具体的な例を含めます：

```markdown
- **レート制限**: [1分あたり○○リクエストなど]
```

## 質問がある場合

不明な点がある場合は、遠慮なくIssueで質問してください。
コミュニティメンバーがサポートします。

- 建設的なフィードバックを提供する
- 他の貢献者を尊重する
- オープンで歓迎的な環境を維持する

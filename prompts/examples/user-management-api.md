# 基本的なAPI仕様書生成プロンプト - ユーザー管理API例

以下の情報に基づいて、詳細なAPI仕様書を生成してください。

## API概要
- **API名**: User Management API
- **バージョン**: v1.0.0
- **ベースURL**: https://api.example.com/v1
- **認証方式**: Bearer Token (JWT)

## エンドポイント一覧

### エンドポイント1: ユーザー一覧取得
- **メソッド**: GET
- **パス**: /users
- **説明**: 登録されているユーザーの一覧を取得する
- **リクエストパラメータ**: 
  - page (integer, 任意): ページ番号（デフォルト: 1）
  - limit (integer, 任意): 1ページあたりの件数（デフォルト: 20、最大: 100）
  - sort (string, 任意): ソート順（created_at, updated_at, name）
- **レスポンス**: ユーザーオブジェクトの配列とページネーション情報

### エンドポイント2: ユーザー詳細取得
- **メソッド**: GET
- **パス**: /users/{userId}
- **説明**: 指定したIDのユーザー情報を取得する
- **リクエストパラメータ**: 
  - userId (string, 必須): ユーザーID（パスパラメータ）
- **レスポンス**: ユーザーオブジェクト

### エンドポイント3: ユーザー作成
- **メソッド**: POST
- **パス**: /users
- **説明**: 新しいユーザーを作成する
- **リクエストパラメータ**: なし
- **リクエストボディ**:
  - name (string, 必須): ユーザー名（3-50文字）
  - email (string, 必須): メールアドレス
  - password (string, 必須): パスワード（8文字以上）
  - role (string, 任意): ユーザーロール（デフォルト: user）
- **レスポンス**: 作成されたユーザーオブジェクト

### エンドポイント4: ユーザー更新
- **メソッド**: PUT
- **パス**: /users/{userId}
- **説明**: 指定したユーザーの情報を更新する
- **リクエストパラメータ**: 
  - userId (string, 必須): ユーザーID（パスパラメータ）
- **リクエストボディ**:
  - name (string, 任意): ユーザー名
  - email (string, 任意): メールアドレス
  - role (string, 任意): ユーザーロール
- **レスポンス**: 更新されたユーザーオブジェクト

### エンドポイント5: ユーザー削除
- **メソッド**: DELETE
- **パス**: /users/{userId}
- **説明**: 指定したユーザーを削除する
- **リクエストパラメータ**: 
  - userId (string, 必須): ユーザーID（パスパラメータ）
- **レスポンス**: 204 No Content

## 生成してほしい内容

以下の項目を含む詳細なAPI仕様書を生成してください：

1. **概要セクション**
   - APIの目的：ユーザー情報の管理（CRUD操作）
   - 対象ユーザー：社内アプリケーションの開発者
   - 主な機能：ユーザーの作成、取得、更新、削除

2. **認証・認可**
   - 認証方法：JWTトークンをAuthorizationヘッダーで送信
   - 必要な権限：admin権限が必要な操作と一般ユーザーでも可能な操作を区別
   - トークンの取得方法：/auth/loginエンドポイントで取得

3. **各エンドポイントの詳細**
   - リクエストの詳細（ヘッダー、ボディ、クエリパラメータ）
   - レスポンスの詳細（ステータスコード、レスポンスボディ）
   - エラーレスポンス（400, 401, 403, 404, 409, 422, 500）
   - 使用例（curlコマンドとJavaScript/Python例）

4. **共通仕様**
   - レート制限：1分あたり60リクエスト
   - ページネーション：Link ヘッダーとX-Total-Countヘッダーを使用
   - フィルタリング・ソート：クエリパラメータで指定

5. **エラーハンドリング**
   - エラーコード一覧（標準的なHTTPステータスコード）
   - エラーメッセージの形式：JSON形式で error, message, details を含む

6. **ベストプラクティス**
   - 推奨される使用方法：適切なページネーションの使用
   - 注意事項：パスワードは平文で取得できないこと、削除は論理削除であることなど
